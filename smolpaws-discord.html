SmolPaws Discord Entry Point live

How the Discord ingress receives messages and dispatches to the agent server.
Source: apps/discord/

Overview

The Discord ingress is a long-running local Node.js process. It connects to Discord via the Gateway WebSocket (discord.js), receives messages, and dispatches them to the shared SmolPaws agent server over HTTP.

flowchart LR
    A["Discord\nGateway"] -->|WebSocket| B["apps/discord\n(discord.js)"]
    B -->|HTTP POST| C["agent-server\n127.0.0.1:8788"]
    C -->|runs agent| D["OpenHands\nruntime"]
    D -->|turn result +\noutbound msgs| C
    C -->|poll + claim| B
    B -->|reply| A

Message Flow

sequenceDiagram
    participant U as Discord User
    participant D as Discord API
    participant I as apps/discord
    participant S as agent-server
    participant A as Agent Runtime

    U->>D: message or @mention
    D->>I: MessageCreate (WebSocket)
    I->>I: shouldRespond() + isAllowed()
    I->>I: buildConversationId()
    I->>I: extractPrompt()
    I->>S: submitConversationMessage()
    S->>A: create/resume conversation
    loop poll until terminal
        I->>S: getTurnStatus()
        I->>S: claimTurnOutboundMessages()
        I-->>D: deliver outbound messages
    end
    I->>S: getTurnResult()
    I->>D: reply in thread
    D->>U: sees reply

Entry Decision Logic

The bot decides whether to respond with two checks:

shouldRespond()

1. Ignore bot messages
2. Respond if bot is @mentioned
3. Respond if trigger pattern matches (default: @smolpaws)
4. Respond to all DMs
5. Otherwise: ignore

isAllowed()

1. If ALLOWED_USERS set → user must be in the list
2. DMs always pass (after user check)
3. If ALLOWED_GUILDS set → guild must be in the list
4. If ALLOWED_CHANNELS set → channel must be in the list

Conversation ID Generation

Stable IDs derived from Discord's own identifiers:

ContextPatternExample
DMdiscord-dm-{author_id}discord-dm-123456789
Threaddiscord-thread-{thread_id}discord-thread-987654321
Channeldiscord-channel-{channel_id}discord-channel-111222333
  • Each DM user gets one persistent conversation
  • Each Discord thread maps to one agent conversation
  • Non-threaded channel messages share a conversation per channel

Turn API Dispatch

The Discord app uses the shared turnClient.ts — the same client used by the GitHub Cloudflare Worker.

flowchart TB
    subgraph "apps/discord"
        A["agentServerClient.ts"] --> B["dispatchToAgentServer()"]
    end
    subgraph "src/shared/turnClient.ts"
        C1["submitConversationMessage()"]
        C2["monitorTurn()"]
        C3["getTurnStatus()"]
        C4["claimTurnOutboundMessages()"]
        C5["getTurnResult()"]
        C2 --> C3 & C4 & C5
    end
    B --> C1
    B --> C2

Key details:

  • deliveryOwnerId — random UUID per dispatch. First submitter wins delivery ownership. Only the owner polls for and claims outbound messages.
  • idempotencyKey — set to Discord message.id to prevent double-processing.
  • createConversation — sent on every submit. Server creates if new, appends if existing.
  • Outbound messages — claimed during polling and delivered to Discord immediately (not batched).

Process Model

  • Single Node.js process, ~290 lines total
  • Uses discord.js Gateway WebSocket (not HTTP webhooks)
  • Typing indicator refreshed every 8 seconds while the agent works
  • Graceful shutdown on SIGINT / SIGTERM
  • Messages > 2000 chars split at natural boundaries
