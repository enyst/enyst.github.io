SmolPaws Slack Entry Point planned

Socket Mode architecture for the Slack ingress — replacing Chrome browser polling with a real Slack app.
Tracking: issue #105 · Source: apps/slack/ (scaffold only)

Why Socket Mode

Three approaches exist for connecting an agent to Slack:

flowchart TB
    subgraph poll ["1. Polling"]
        direction LR
        P1["Cron job"] -->|every 60s| P2["conversations.history"] --> P3["check triggers"] --> P4["start conversation"] --> P5["chat.postMessage"]
    end
    subgraph hooks ["2. HTTP Webhooks"]
        direction LR
        W1["Slack"] -->|POST| W2["Public URL"] --> W3["verify + process"] --> W4["chat.postMessage"]
    end
    subgraph socket ["3. Socket Mode ✓"]
        direction LR
        S1["Slack"] <-->|WebSocket| S2["Local process"] --> S3["process"] --> S4["chat.postMessage"]
    end

SmolPaws uses Socket Mode because:

  • No public URL or tunnel required — the bot connects outbound to Slack
  • Event-driven, not poll-driven — no timestamp drift, no missed windows
  • Fits the existing pattern: Discord and WhatsApp are both long-running local processes
  • Fewer API scopes needed — the app reacts to delivered events rather than scraping history

Target Architecture

flowchart LR
    A["Slack\nSocket Mode"] <-->|WebSocket| B["apps/slack\n(@slack/bolt)"]
    B -->|HTTP POST| C["agent-server\n127.0.0.1:8788"]
    C -->|runs agent| D["OpenHands\nruntime"]
    D -->|turn result +\noutbound msgs| C
    C -->|poll + claim| B
    B -->|chat.postMessage\nwith thread_ts| A

Same thin-ingress pattern as Discord:

  1. Receive Slack event over Socket Mode WebSocket
  2. Build a stable conversation ID from Slack identifiers
  3. Dispatch to the agent server via the shared turnClient.ts
  4. Poll for turn completion and claim outbound messages
  5. Deliver results back to Slack in the correct thread

Event Handling

Phase 1 handles two event types:

app_mention

Fires when someone types @smolpaws in a channel. The bot replies in a thread under the mentioning message.

app.event("app_mention")
  → extract text, strip bot mention
  → build conversation ID from team + channel + thread_ts
  → dispatch to agent server
  → reply with thread_ts = event.ts (or event.thread_ts if already threaded)

message.im

Fires on every DM to the bot. No mention required.

app.event("message")  // filter: channel_type === "im"
  → skip bot messages and subtypes
  → build conversation ID from team + channel
  → dispatch to agent server
  → reply in the DM

Conversation ID Generation

Stable IDs scoped to the Slack workspace:

ContextPatternExample
DMslack-im-{team_id}-{channel_id}slack-im-T06P-D08X
Channel threadslack-thread-{team_id}-{channel_id}-{thread_ts}slack-thread-T06P-C06P-1717200000.000100

Design decisions:

  • DMs use channel_id, not user_id — Slack assigns a unique DM channel per user pair, equivalent but consistent with Slack's model
  • Threads always use thread_ts — the bot always replies in a thread, using the root message's ts as thread_ts
  • team_id included — supports multi-workspace without collision

Comparison With Hermes

Hermes has a mature Slack adapter (~3000 lines, Python). The two designs target different tradeoffs:

        H1 --> H4["DiscordAdapter"]
        H1 --> H5["WhatsAppAdapter"]

SmolPaws' design is intentionally simpler. The complexity lives in the agent server, not the ingress adapter. The Hermes adapter is a useful reference for what SmolPaws might eventually need.

Open Questions

  1. Channel-thread history — Should the bot fetch bounded thread context? Adds scopes and complexity. Leaning no for MVP.
  2. Slash commands — Hermes shows the pattern (regex dispatcher, ephemeral ack, response_url replace). Leaning wait.
  3. LaunchAgent — Ship as manual npm run dev first, or include a LaunchAgent from day one? Leaning manual first.
  4. Dedup — Socket Mode can redeliver events on reconnect. Simple in-memory set with TTL should suffice.
